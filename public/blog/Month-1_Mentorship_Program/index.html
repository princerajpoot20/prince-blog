Designing the \help Endpoint: Key Decisions

I began thinking about the structure of the /help endpoint. Given that our objective is to provide help for multiple endpoints, let‚Äôs start by considering how we would pass data to the /help endpoint.
The project idea already suggests utilizing the URL approach, as indicated by help/{command}.
You might be wondering why the project idea leaned in that direction, especially when the request body is such a viable option. Let‚Äôs first understand the differences between the two, and then delve into why we chose URL over the request body.

  • URL-based data transmission is ideal when:

    • The data amount is small, ensuring URLs don‚Äôt get too lengthy.
    • The information isn‚Äôt sensitive. For context, transmitting sensitive data, like passwords, in URLs can expose them.

  • Request body data transmission becomes crucial when:

    • Data size exceeds the URL‚Äôs character limit (usually around 2048 characters).
    • There‚Äôs a need to send confidential information, which won‚Äôt be visible in the URL.

Considering both the non-sensitive nature of the data and its compactness, using the URL method is the best choice.
